# AGENTS.md

## Overview

ResumeLite is now a local-only pnpm workspace for personal resume editing. `apps/web` is the only runnable app. It is built with React 19, Vite, TanStack Router, TanStack Query, Zustand, Tailwind CSS, and shared resume/export packages.

There is no standalone backend server in this light version. The Vite app exposes only a local `/api/local/*` middleware for file-backed resume persistence. Do not reintroduce login, registration, OAuth, API keys, Better Auth, oRPC server routes, MCP, PostgreSQL, Drizzle migrations, Docker Compose, hosted public sharing, or account settings unless the user explicitly asks for a new server-backed product direction.

## Prerequisites

- Node.js 24.
- pnpm 11.x through Corepack.
- No Docker, PostgreSQL, Redis, S3, SMTP, or auth secret is required for normal local use.

## Codebase Map

- `apps/web` owns routes, dashboard, builder UI, preview, import/export controls, local persistence, i18n, theme, and app-level state.
- `apps/web/src/features/resume/local` owns the file-backed local resume repository and local API adapter. New resume CRUD should go through this module.
- `packages/schema` contains Zod schemas and typed resume/page/template models.
- `packages/resume` contains pure resume-domain helpers.
- `packages/pdf` contains React PDF rendering and browser PDF generation.
- `packages/docx` contains DOCX export generation.
- `packages/import` contains resume importers.
- `packages/fonts` contains font metadata.
- `packages/ui` contains shared Base UI/shadcn-style primitives and hooks.
- `packages/utils` contains narrow cross-cutting utilities with explicit export subpaths.
- `packages/config` contains shared development configuration.

## Local Persistence

Resumes are stored in `data/local/resumes.json` through the local Vite API. The local repository mirrors the original resume service semantics where useful: metadata list queries, full resume reads, slug uniqueness, tags, created/updated timestamps, lock protection for update/delete, and atomic serialized writes.

Resume and experience writes never use `localStorage` or IndexedDB — the file-backed local API is the only store.

Project experiences are stored the same way in `data/local/experiences.json` through `/api/local/experiences`, and `apps/web/src/features/project/local` owns that repository (parallel to the resume one).

## AI-assisted resume writing (any agent)

This project has no in-app LLM. Instead, the coding agent (Claude Code, Codex, or any other) is the "AI": it reads the local data and writes resumes through the local API. The canonical workflow lives in **`.claude/skills/resumelite-workspace/SKILL.md`** — Claude Code loads it automatically as a skill; **other agents should read that file directly** and follow it.

In short: read `data/local/experiences.json` (the factual project-experience library) and the resume schema in `packages/schema/src/resume`, then create/update resumes via `/api/local/resumes` (duplicate a base resume for style, then PATCH its `data`). Draw only on facts already in the experience library — never invent projects, employers, dates, or metrics. The running app refreshes live when the data files change (SSE at `/api/local/events` + a `data/local` file watcher). For the resume-writing craft itself, Claude Code users compose with the `resume-tailor` / `tailored-resume-generator` skills; agents without those skills draft the content directly using good judgment.

```bash
pnpm --filter web typecheck

pnpm --filter web build
```

## Boundaries

- Workspace dependencies must go through package names and package export maps.
- Do not import another workspace's private `src` tree through repository paths, `@resumelite/*/src/*`, or TypeScript path aliases.
- Browser-only app workflows belong in `apps/web/src/routes` or `apps/web/src/features`.
- Pure resume data behavior belongs in `packages/resume`.
- Resume schema changes start in `packages/schema`.
- PDF rendering belongs in `packages/pdf`.
- DOCX generation belongs in `packages/docx`.
- Shared UI primitives belong in `packages/ui`; workflow-specific UI stays in the owning web feature.

## Gotchas

- Do not hand-edit `apps/web/src/routeTree.gen.ts`; it is generated by TanStack Router tooling.
- The app should open directly to local dashboard/editor flows without requiring a session.
- `/auth`, `/agent`, `/dashboard/settings`, public username/slug routes, hosted `/api` routes, and `/mcp` are intentionally removed. The only API surface is local-only `/api/local/*`.
- `pnpm check` is write-capable because it runs `biome check --write --unsafe .`; use `pnpm exec biome check .` for read-only inspection.
- There may be unrelated local edits in the worktree. Inspect `git status --short` first and do not revert changes you did not make.
